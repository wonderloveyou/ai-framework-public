# User Manual RU — Skill

## Goal
Написать руководство пользователя, которое реально помогает человеку сделать нужное действие без саппорта и гуглинга. Руководство нужно не для того, чтобы показать крутость продукта — оно нужно, чтобы пользователь перестал его читать и начал работать.

## Inputs
- Продукт / сервис (что описываем)
- Целевая аудитория (новички, продвинутые, админы — или всё сразу)
- Тип руководства: сценарное (по задачам) / описательное (по функциям) / смешанное
- Формат: help-центр (web) / PDF / встроенные подсказки / чат-бот

## Outputs
- Руководство / инструкция / статья в help-центр
- Структура: от быстрого старта до ошибок и FAQ

## Workflow

### Phase 0: Research — до написания

**A. User Research**
Прежде чем писать — собери данные:
1. **Метод «Top 10 tickets»** — выгрузи последние 90 дней обращений в поддержку, сгруппируй по причинам, отсортируй по частоте. Топ-10 — это оглавление твоей документации. Если после прочтения человек всё равно пишет в поддержку — руководство не работает.
2. **User Personas** — создай 2-3 профиля читателей: новичок (первый раз), обычный пользователь (ежедневные задачи), админ (настройки, права, безопасность). Пиши для каждого свой раздел, не смешивай.
3. **User Journey Map** — пройди путь пользователя с первого входа до ключевых сценариев. На каждом шаге запиши: что он видит, что хочет сделать, где может застрять. На стыках самых частых застреваний пиши статьи.
4. **Конкурентный анализ** — посмотри как аналогичные продукты пишут документацию. Отметь что работает, что нет.

**B. Определение объёма**
Ответь до написания:
1. Кто читает? (новичок / продвинутый / админ / технарь)
2. В какой ситуации он открыл руководство? (проблема / непонимание / интерес)
3. Какую конкретную задачу он хочет решить прямо сейчас?
4. Какой формат ему удобен? (пошаговая инструкция / справочник / FAQ)
5. Где он будет читать? (дома / на работе / в машине — определяет длину и формат)
6. Как часто он будет обращаться к руководству? (один раз / регулярно — определяет глубину)

**Золотое правило:** не смешивай уровни в одной статье. Быстрый старт — отдельно, расширенные настройки — отдельно.

### Phase 1: Выбери подход

**Сценарный** — когда пользователь приходит с задачей:
«Как выложить трек», «Как вывести деньги», «Как сбросить пароль»
- В основе — задачи пользователя, не структура продукта
- Каждый раздел = один сценарий
- Пошаговая инструкция: 1-2 действия на шаг

**Описательный** — когда нужно объяснить сложный интерфейс:
«Что делает кнопка "Сброс"», «Какие есть форматы экспорта»
- В основе — элементы интерфейса
- Полезно, когда действие кнопки неочевидно или есть нюансы
- Описывает состояния: однократное / двойное / длинное нажатие

**Смешанный** — для сложных продуктов:
- Сценарные разделы для ключевых задач
- Описательный справочник для настроек и параметров

### Phase 2: Выбор архитектуры

Есть 4 способа организовать контент:

**1. По задачам (сценарная)** — лучшая для help-центра.
Каждый раздел = задача пользователя: «Как выложить трек», «Как вывести деньги».
Контент группируется по реальным сценариям, а не по меню продукта.

**2. По функциям (модульная)** — для описательных руководств.
Разделы соответствуют модулям продукта: «Раздел Релизы», «Раздел Выплаты». Работает когда функциональность сложная и неочевидная.

**3. По типам информации** — для大型 документации.
- **Концептуальная** — как устроен продукт, основные понятия
- **Инструктивная** — пошаговые руководства
- **Справочная** — кнопки, параметры, API, форматы
Каждый тип — свой раздел, чтобы пользователь не путался

**4. По фазам использования** — для сложных продуктов с онбордингом.
- **Onboarding** — первый вход, быстрый старт
- **Daily** — ежедневные задачи
- **Admin** — настройки, права, интеграции
- **Troubleshooting** — ошибки, проблемы

**Progressive Information Disclosure:** начинай с самого простого и короткого пути. Пользователь получает essentials за 2 минуты. Если нужно больше — рядом ссылка «Подробнее» с полной информацией. Это уменьшает страх перед документацией.

### Phase 3: Структура руководства

Рекомендуемая структура help-центра (по фазам использования):

```
1. Быстрый старт (5 минут)
   - Минимальный путь до первого результата
   - 1-2 примера с скринами

2. Основные сценарии (80% задач)
   - Группировка по ролям (артист, лейбл, менеджер)
   - Или по категориям (релизы, выплаты, отчёты)
   - Каждый сценарий = отдельная статья

3. Подробные инструкции
   - Пошаговые гайды по функциям
   - Формат: цель → что нужно заранее → шаги → результат → если не получилось

4. Ошибки и проблемы
   - Ошибка → причина → что делать → кому писать
   - Собирать из реальных обращений в поддержку

5. FAQ
   - Короткие ответы на частые вопросы
   - Обновлять после каждого релиза

6. Глоссарий (если есть сложные термины)

7. Поддержка
   - Куда писать, какие данные прикладывать
```

### Phase 4: Формат статьи (шаблон)

```md
# Как [задача пользователя]  (не «Настройки» — а «Как вывести деньги»)

## Цель
Что получится после выполнения. (1 предложение)

## До начала
Что нужно иметь/знать заранее (права доступа, подписка, данные).

## Инструкция
1. Перейдите в [раздел] → [подраздел].
2. Нажмите [кнопка].
3. Заполните [поле] — вот пример: ...
4. Нажмите [Сохранить].

## Результат
Что изменилось. Что пользователь увидит на экране.

## Если не получилось
- Ошибка «Текст ошибки»: причина → решение
- Если кнопка неактивна: что проверить
- Если нет нужного раздела: какие права нужны
```

**Правила для шагов:**
- **Один шаг — одно действие.** Не пиши «откройте раздел и введите данные» в одном шаге
- **Повелительное наклонение:** «Нажмите», «Выберите», «Введите»
- **Не больше 2 предложений на шаг.** Если больше — разбей
- **7-9 шагов максимум** на одну процедуру. Если больше — логически разбей на подпроцедуры
- **Parallel structure:** все шаги начинаются одинаково (с глагола), в списках — одинаковая грамматика
- **Context first:** перед шагами дай контекст — когда и зачем это делается
- **Confirmation:** после выполнения покажи, что пользователь увидит на экране

### Phase 5: Readability — проверка читаемости
Каждую статью проверяй по этим параметрам:
- **Длина предложения:** 15-20 слов максимум. Если предложение длиннее 30 слов — разбивай на два
- **Абзацы:** 3-5 предложений максимум. Один абзац = одна мысль
- **Однородные списки:** в маркированном списке все пункты должны начинаться с одинаковой части речи. «Откройте», «Нажмите», «Введите» — хорошо. «Откройте», «После этого нажимаем», «Должно открыться окно» — плохо
- **Исключи лишние слова:** «После этого необходимо перейти в» → «Перейдите в». «Для того чтобы» → «Чтобы». «В настоящее время» → «сейчас» (или выкинуть)
- **Прочитай вслух.** Если спотыкаешься — перепиши. Это лучший тест качества

### Phase 6: Стиль и язык

**Пиши:**
- Просто и разговорно, без бюрократического шума
- «Войдите в аккаунт» вместо «Произведите авторизацию посредством ввода идентификационных данных»
- Активный залог: «Система обновит данные» вместо «Данные будут обновлены»
- Добавляй последствия: «Нажмите Сохранить. Изменения применятся сразу и будут видны всем участникам»
- Единую терминологию: «релиз», «релиз», а не «релиз/релиз записи/проект»

**Не пиши:**
- Очевидное: «Кнопка Сохранить используется для сохранения» — мусор
- Маркетинг: руководство не лендинг
- «Как видно из рисунка 3.2.1б» — это неуважение
- Обороты вроде «Рекомендуется нажать», «Необходимо осуществить»

### Phase 7: Визуал

- Скриншот — только для сложных мест. Не вставляй 10 скринов подряд
- На скрине — только нужная область, не весь экран
- Выделяй важный элемент рамкой или стрелкой
- GIF/видео — для сложных анимаций, до 2 минут, по запросу
- Схема или flow chart лучше текста для процессов

### Phase 8: Обучай в процессе

Лучшие руководства не просто дают шаги — они обучают пользователя:

**Плохо:** «Загрузите цветное бельё (не более 5 кг)»

**Хорошо:** «Загрузите цветное бельё (не более 5 кг). Проверьте, чтобы вещи можно было стирать на 40°. Температурный режим стирки указан на ярлычке.»

**Плохо:** «Придумайте пароль»

**Хорошо:** «Придумайте пароль для доступа к беспроводной сети (8-12 знаков). По этому паролю вы и ваши гости будут подключаться к интернету. Желательно, чтобы он не совпадал с паролем от вашей почты.»

### Phase 9: Ошибки и предупреждения

**Плохо:** «Будьте внимательны»

**Хорошо:** «После удаления проекта восстановить его нельзя. Перед удалением экспортируйте данные.»

Формат для типовых ошибок:
```
Ошибка: «Недостаточно прав»
Причина: у пользователя нет роли администратора
Что делать: попросить владельца проекта выдать доступ
Кому писать: если владелец неизвестен — обратиться в поддержку
```

### Phase 10: Doc-as-Code и актуальность

Документация должна жить как код: версионироваться, ревьюиться, тестироваться.

- **Sync with releases.** Если интерфейс изменился в пятницу — обнови руководство к понедельнику. Устаревшая инструкция хуже, чем её отсутствие
- **Версионирование.** Если поддерживаешь несколько версий продукта — веди документацию для каждой. Укажи на каждой странице версию продукта и дату обновления
- **Review цикл.** Перед публикацией: автор → тех. ревью (разработчик проверяет точность) → UX-ревью (дизайнер проверяет скрины) → редактура
- **Feedback loop.** Кнопка «Помогло / Не помогло» + «Сообщить об ошибке» внизу каждой страницы. Раз в месяц смотри analytics — какие статьи ищут, но не находят

### Phase 11: Метрики качества документации

Руководство должно измеряться, а не просто «быть написанным»:

| Метрика | Норма | Как измерить |
|---------|-------|-------------|
| Время до первого успешного действия | < 2 минут | Дать новому пользователю задачу + засечь |
| Процент решивших проблему без поддержки | > 80% | Сравнить число обращений по теме до и после статьи |
| TTV (Time to Value) | < 5 минут | От первого входа в help до выполнения ключевого действия |
| Search success rate | > 90% | Аналитика поиска по документации — находят ли то что ищут |
| Статья помогла / не помогла | > 70% «помогло» | Кнопка feedback под каждой статьёй |

- Ставь дату обновления и версию продукта на каждой странице
- Если интерфейс изменился в пятницу — обнови руководство к понедельнику
- Устаревшая инструкция хуже, чем её отсутствие
- Кнопка «Помогло / Не помогло» внизу каждой страницы

## Чеклист качества

### До публикации
- [ ] Заголовок совпадает с задачей пользователя («Как вывести деньги», не «Настройки платежей»)
- [ ] Понятно за 3 секунды — для чего этот раздел
- [ ] Есть пошаговая инструкция (1-2 действия на шаг, 7-9 шагов макс)
- [ ] Нет лишней теории — только то что нужно для задачи
- [ ] Показан результат: что изменится, что увидит на экране
- [ ] Описаны типовые ошибки с причинами и решениями
- [ ] Указаны ограничения: права доступа, тарифы, роли
- [ ] Parallel structure: все шаги с одинаковой грамматикой
- [ ] Терминология единая по всему руководству (глоссарий)
- [ ] Предложения 15-20 слов, абзацы 3-5 предложений
- [ ] Скриншоты актуальны, показывают только нужную область
- [ ] Прочитано вслух — нигде не спотыкаешься

### После публикации
- [ ] Дано почитать новичку + засечено время до первого действия (< 2 мин)
- [ ] Кнопка «Помогло / Не помогло» под статьёй
- [ ] Дата и версия продукта на каждой странице
- [ ] Собрана первая неделя analytics — какие темы ищут но не находят

## Hard rules
- Не пиши «как видно из рисунка», «как было сказано ранее» — это неуважение к читателю
- Не используй маркетинг и рекламные формулировки — руководство не лендинг
- Не описывай интерфейс — описывай решение задачи
- Не пиши одной простынёй — один раздел = одна задача
- Не объясняй через цвет («нажмите зелёную кнопку») — напиши название и цвет в скобках
- Каждый шаг — одно действие в повелительном наклонении
- Ошибки описывай с причиной и решением, а не «обратитесь к администратору»
- Не смешивай концептуальную, инструктивную и справочную информацию в одном разделе
- Не объясняй через цвет — пиши название элемента и цвет в скобках
- Не больше 9 шагов на процедуру — длинные разбивай на подпроцедуры
- Все пункты списка должны иметь одинаковую грамматическую структуру
- Если интерфейс изменился — обнови документацию до или синхронно с релизом, не после
