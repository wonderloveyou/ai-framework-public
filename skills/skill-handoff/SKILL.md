# Handoff Writer

## Contract

| Aspect | Value |
|--------|-------|
| **Input** | Session history (already in context), optional topic/request from $ARGUMENTS |
| **Output** | Dated `HANDOFF-YYYY-MM-DD.md` in project root + copy in Obsidian vault |
| **Side effects** | Creates new file. Never overwrites previous handoffs. Never modifies code. |
| **Guard** | Skip if session was purely conversational with no code work |

## Anti-Rationalization

| Excuse | Reality |
|--------|---------|
| "Сессия была короткой, handoff не нужен" | Короткие сессии теряются быстрее всего. 2 минуты на handoff экономят 20 минут на восстановление контекста. |
| "Я и так помню что делал" | Через 3 дня и 5 сессий ты не помнишь. Следующий агент — тем более. |
| "Обновлю старый HANDOFF.md" | Overwrite убивает историю. Датированные файлы накапливаются и не теряются. |
| "Obsidian копия необязательна" | Obsidian — твой второй мозг. Handoff в vault доступен даже когда проект закрыт. |
| "Опущу детали, главное — итог" | Детали решают. «Поменял auth.ts» бесполезно. «Поменял auth.ts:42 — убрал race condition в token refresh» — полезно. |

## Mode Detection

| Trigger | Mode | What happens |
|---------|------|-------------|
| "быстрый handoff", "quick handoff", "короткий handoff", "снапшот" | **quick** | Сжатый снапшот для себя. 30 секунд. Только: текущая цель + что сделано + что дальше. |
| "продолжи handoff", "resume", "resume handoff", "восстанови контекст" | **resume** | Читает последний `HANDOFF-*.md` + входит в контекст. Без записи. |
| "handoff", "подготовь handoff", "передай контекст" (по умолчанию) | **full** | Полный handoff с контекстом проекта, детальным логом, запросом к следующему агенту. |

## Workflow

### Mode: resume

1. Найди `HANDOFF-*.md` в корне проекта
2. Прочитай последний по дате
3. Если есть ссылки на предыдущие — прочитай и их
4. Выведи: «Контекст восстановлен из N handoff-файлов. Последняя сессия: <дата> — <тема>. Готов к работе.»
5. Жди задачу пользователя

Ничего не записывай. Это read-only операция.

### Mode: full (по умолчанию)

### Phase 1: Собери контекст

Прочитай и зафиксируй:

1. **Файлы проекта** — `CLAUDE.md`, `AGENTS.md`, `program.md`, `CONTEXT.md`, `ARCHITECTURE.md` (если есть)
2. **Git статус** — `git status --short`, `git log --oneline -5`, `git diff --stat`
3. **Ключевые файлы** — которые менялись в этой сессии

### Phase 2: Зафиксируй сделанное

По шагам, детально:
- Какие файлы изменены/созданы и что именно
- Какие решения приняты и почему
- Какие баги найдены и исправлены
- Что проверено (тесты, линтер, typecheck)
- Что осталось незавершённым

### Phase 3: Напиши датированный handoff

**Критично: НИКОГДА не перезаписывай старые handoff-файлы.**

Формат имени: `HANDOFF-YYYY-MM-DD.md` (сегодняшняя дата).

Сохрани в корень проекта. Если уже существует файл с этой датой — добавь суффикс: `HANDOFF-YYYY-MM-DD-2.md`.

Структура:

```markdown
# Handoff — YYYY-MM-DD — <тема сессии>

## Инструкция для следующего агента

1. **Прочитай этот файл полностью.** Это твой контекст.
2. **Найди предыдущие handoff** — `HANDOFF-*.md` в корне проекта, прочитай последний (по дате).
3. **Проверь структуру проекта** — ключевые папки и файлы.
4. **Посмотри git log и diff** — что менялось.
5. **Прочитай CLAUDE.md** — правила проекта.
6. **Сделай code review изменений** — свежим взглядом.
7. **Выдай фидбек** — что хорошо, что плохо, что переделать.

## Контекст проекта

**Проект:** <название>
**Стек:** <языки, фреймворки, БД>
**Ключевые папки:**
- `src/` — <что там>
- ...

**Ключевые файлы:**
- `CLAUDE.md` — <где>
- `program.md` — <где>

**Предыдущие handoff:**
- `HANDOFF-YYYY-MM-DD.md` — <тема той сессии>
- ...

## Что сделано

### Шаг 1. <заголовок>
**Задача:** <что делали>
**Файлы:** <какие изменены, с путями>
**Решения:** <что решили и почему>
**Результат:** <что получилось, проверки>

### Шаг 2. ...

## Текущий статус

**Готово:**
- <пункт>

**Осталось:**
- <пункт>

**Известные проблемы:**
- <проблема> → <статус>

## Запрос к следующему агенту

<конкретная просьба: "проверь безопасность эндпоинтов", "найди почему падает тест X", "оцени архитектуру модуля Y">

Будь безжалостно честным. Лучше найти проблему сейчас, чем в проде.
```

### Phase 4: Сохрани и сообщи

1. Сохрани `HANDOFF-YYYY-MM-DD.md` в корень проекта
2. Если есть Obsidian vault — сохрани копию в `Jarvis/01_raw/sessions/handoff-YYYY-MM-DD.md`
3. Сообщи пользователю: список всех handoff-файлов в проекте и что нового

### Mode: quick

Лёгкий снапшот. Когда нужно быстро зафиксировать состояние для себя, без детального контекста для другого агента.

Структура (20-30 строк, не больше):

```markdown
# Quick — YYYY-MM-DD HH:MM — <тема>

**Цель:** <1 предложение — что делаем в этом проекте>
**Сделано:** <3-5 буллетов — что конкретно>
**Дальше:** <1-3 буллета — что осталось>
**Не работает:** <что сломано или заблокировано>
**Файлы:** <список изменённых файлов>
```

Сохрани в `HANDOFF-YYYY-MM-DD.md` (такой же формат имени, но с пометкой `Quick` в заголовке). Не дублируй в Obsidian — quick-снапшоты только в проекте.

## Цепочка сессий

```
HANDOFF-2026-06-10.md   ← сессия 1 (сохранён)
HANDOFF-2026-06-11.md   ← сессия 2 (сохранён)
HANDOFF-2026-06-12.md   ← сессия 3 (сохранён, ссылается на 06-11)
```

Новый агент:
1. Ищет `HANDOFF-*.md` в корне
2. Читает последний по дате
3. Видит ссылки на предыдущие — может провалиться глубже если нужно
4. Входит в контекст за 30 секунд

## Outputs

1. `HANDOFF-YYYY-MM-DD.md` в корне проекта (новый файл, старые не тронуты)
2. Копия в Obsidian vault (если доступен)
3. Сообщение: список всех handoff в проекте

## Hard rules

- **Никогда не overwrite.** Каждая сессия = новый файл с датой.
- **Ссылайся на предыдущие.** В секции «Предыдущие handoff» перечисли файлы с датами и темами.
- **Пиши максимально подробно.** Следующий агент не видел твою сессию.
- **Конкретные пути к файлам.** Агент должен найти без подсказок.
- **Инструкция — первый раздел.** Агент знает что делать ДО погружения в контекст.
- **Не оценивай свою работу.** Пусть свежий агент сделает ревью.
