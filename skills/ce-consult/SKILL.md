# ce-consult — LLM Council Context Packager

Based on Andrej Karpathy's LLM Council technique: a fresh LLM with full context catches blind spots the current agent misses. Even if the current session still has context space, a different model (or same model in a fresh session) provides genuinely independent review.

This skill PACKAGES context. It does not call external APIs. The user copies the output and pastes it wherever they want — Claude, GPT, Gemini, or multiple models in parallel.

## Contract

| Aspect | Value |
|--------|-------|
| **Input** | Current session context, user's description of the blocker/question |
| **Output** | `CONSULT-YYYY-MM-DD.md` — self-contained, ready to paste into any LLM |
| **Side effects** | Creates file in project root. Read-only otherwise. |
| **Guard** | Do not consult for trivial questions. The council is for hard problems. Never ask the reviewer to implement — only to analyze. The reviewer is a critic, not a builder. |

## Anti-Rationalization

| Excuse | Reality |
|--------|---------|
| "Я и так всё проверил, другой LLM не найдёт нового" | Свежий взгляд находит то, что текущий агент перестал видеть из-за saturation. Это работает даже с той же моделью. |
| "Это займёт много времени" | Упаковка контекста — 3 минуты. Исправление ошибки, которую найдёт свежий агент — часы. |
| "Достаточно короткого вопроса" | Короткий вопрос без контекста = поверхностный ответ. Контекст — это разница между «выглядит ок» и «вот конкретная проблема на строке 42». |
| "Консул только для кода" | Консул работает для любой задачи: код, текст, стратегия, архитектура, бюрократия. Свежий взгляд универсален. |

## Execution

### Phase 0: Определи scope

Из $ARGUMENTS или диалога определи:
1. **Домен задачи** — код / текст / стратегия / архитектура / другое
2. **Где затык** — по мнению пользователя И по твоему мнению (они могут различаться)
3. **Что уже проверено** — чтобы совет не повторил пройденное
4. **Какой результат нужен** — конкретный ответ / список проблем / альтернативный подход / всё вместе

### Phase 1: Собери контекст проекта

Включи в пакет:

1. **Project identity** — что за проект, цель, стек, аудитория
2. **CLAUDE.md / AGENTS.md** — ключевые правила (не целиком, выжимка)
3. **Архитектура** — если есть, ключевые решения
4. **Текущий статус** — из последнего handoff или git log

### Phase 2: Собери контекст задачи

Детально, без купюр:

1. **Что делали** — пошагово, с конкретными файлами и решениями
2. **Где застряли** — две точки зрения (пользователь + агент)
3. **Что пробовали** — и почему отбросили
4. **Что проверили** — тесты, линтер, ручная проверка
5. **Ключевые фрагменты** — код, текст, схема (не весь файл, релевантные куски)

### Phase 3: Сформулируй вопрос совету

Конкретный запрос. Не «посмотри код». А:

```
- В функции parseConfig() на строке 42 есть race condition?
- Правильно ли разбита архитектура на модули?
- Какой подход к кэшированию выбрать: Redis vs in-memory?
- Где слабые места в этом плане?
- Какие альтернативные решения мы не рассмотрели?
```

И ключевое: **попроси совет быть безжалостно честным.** «Лучше найти проблему сейчас, чем в проде. Не пиши код — только анализ и конкретные рекомендации. Исполнять будет другой агент.»

### Phase 4: Упакуй в CONSULT.md

```markdown
# CONSULT — YYYY-MM-DD — <тема>

> Этот файл создан скиллом ce-consult для внешнего ревью.
> Отправь его содержимое в новый чат с Claude/GPT/Gemini (или нескольким моделям параллельно).
> Свежий LLM с этим контекстом найдёт слепые зоны, укажет на ошибки, предложит альтернативы.
> Техника основана на LLM Council Андрея Карпатого (github.com/karpathy/llm-council).

---

## Инструкция для ревьюера (LLM)

Ты — свежий, независимый ревьюер. У тебя нет предыстории этого диалога.
Ты видишь только то, что в этом файле. Твоя задача:

**🚫 Ты не исполняешь. Ты только анализируешь.**
Не пиши код. Не исправляй текст. Не редактируй файлы. Не предлагай конкретную реализацию строк. Твоя работа — найти проблемы, указать на ошибки, предложить альтернативы на уровне идей. Исполнять будет другой агент.

1. **Прочитай весь файл.** Не пропускай секции. Контекст — это всё.
2. **Найди слепые зоны.** Что автор(ы) не заметили? Какие допущения не проверены?
3. **Укажи на ошибки.** Если видишь логическую дыру, архитектурную проблему, фактическую неточность — скажи прямо.
4. **Предложи альтернативы.** Как ещё можно решить эту задачу? Что не было рассмотрено?
5. **Оцени риски.** Что может пойти не так? Где самые хрупкие места?
6. **Дай конкретные следующие шаги.** Не «подумайте над архитектурой», а «выделите модуль X в отдельный файл, потому что...»

Будь безжалостно честным. Лучше найти проблему сейчас, чем потом.
```

### Phase 5: Сохрани и проинструктируй

1. Сохрани `CONSULT-YYYY-MM-DD.md` в корень проекта
2. Сообщи пользователю:
   - Размер файла
   - Куда копипастить (новый чат Claude / GPT / Gemini)
   - Совет: отправить параллельно в 2-3 модели для максимального эффекта (Karpathy-style)
   - Что делать с ответом: вернуться в этот чат и сказать «вот что сказал консул: ...»

## Советы пользователю

- **Отправь минимум в 2 модели.** Разные модели находят РАЗНЫЕ проблемы. GPT может заметить архитектурную дыру, которую Claude пропустил, и наоборот.
- **Не редактируй CONSULT.md перед отправкой.** Самоцензура убивает смысл. Свежий агент должен видеть всё как есть.
- **Вернись с ответом.** Скажи: «Вот что сказал консул: [вставить ответ]. Что думаешь?» — и текущий агент обработает фидбек.
