## CSS Token Rule

Velmi should override variables and tokens first:

- colors
- fonts
- spacing scale
- radius/shadow intensity
- decorative accents

Do not fork the base layout CSS unless a new format requires it.

## Visual Direction

Serious B2B technical case with poster-level clarity.

Use:

- dark technical base for AI/system topics
- high-contrast white text
- violet/cyan for AI and system mechanics
- red only for pain/error
- orange only for action/punch
- mono labels, grid, small interface marks as support details
- large numbers as visual objects
- compact diagrams over paragraphs

Avoid:

- tiny gray text on dark backgrounds
- metric recaps on the final slide
- sticker overload
- every slide using a different style
- infoproduct gold/burgundy aesthetics
- meme references in serious cases

## Reusable Velmi Patterns

- visual passport: central product + 3-4 facts around it
- big metric: one huge number + short interpretation
- before/after: equal-weight comparison
- LLM vs CODE: responsibility boundary
- metric -> interpretation
- principle -> concrete example

## Refi Visual Findings

The `PICS` folder was reviewed through a contact sheet, not only as filenames. Useful families:

- dark neon/technical posters: strong for hooks, but avoid shock/meme aggression
- white expert objection cards: strong for readable educational slides
- red/orange sales cards: useful for mistake/fix and CTA, risky if overused
- violet grunge/story cards: useful for energy and hierarchy, risky for serious Habr cases
- gold/burgundy premium cards: mostly not Velmi-compatible

Palette findings:

- black/graphite + white/milk + one bright accent works best
- violet/cyan suits AI/mechanism
- red should mean pain/error
- orange should mean action/punch
- pale palettes can support expert cards but should not become the main Velmi look

## Bitrix Lesson

Accepted final principle:

- LLM: dialogue, intent, next question, manager summary
- Code: deduplication, status whitelist, deadlines, confidence thresholds, CRM writes

For AI automation cases, end with a responsibility boundary, not a metrics recap.
