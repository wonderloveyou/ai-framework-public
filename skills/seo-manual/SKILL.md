# Skill: SEO Pipeline — универсальный пайплайн для любого сайта

> Создан на основе проекта Velmi AI (2026-07-01). Применим к B2B, B2C, SaaS, ecom.
> Меняются только ключи и офферы — механика одна.

## Goal

За 90 дней построить SEO-актив: главная + лендинги + блог + кейсы + лид-магниты. Без ссылок. Через интенты и реальные формулировки пользователей.

## Inputs

- Продукт/услуга, офферы, ЦА (из CONTEXT.md)
- Текущий сайт (если есть): URL, CMS, robots.txt, sitemap.xml
- Доступ к GSC, Яндекс.Вебмастер, Яндекс.Метрика, GA4
- Yandex Cloud аккаунт с платёжным аккаунтом (для Wordstat API)

## Outputs

- Семантическое ядро (CSV + JSON)
- Технический baseline сайта
- Мини-брифы на каждую страницу
- Готовые тексты: главная, лендинги, статьи, кейсы, лид-магниты
- Схема перелинковки + E-E-A-T
- GSC-итерации с правками

## Workflow

### Фаза 0: Подготовка проекта (день 0)

- Создаём `CONTEXT.md` — продукт, аудитория, ограничения
- Создаём `AGENTS.md` — hard rules для всех агентов
- Фиксируем офферы: что продаём, кому, по чём
- **Технический аудит текущего сайта** (если уже есть):
  - robots.txt, sitemap.xml, индексация (`site:domain.com`)
  - PageSpeed / Core Web Vitals (Lighthouse) — baseline
  - Мобильная адаптация (Mobile-Friendly Test)
  - Битые ссылки, дубли, 404 (Screaming Frog до 500 URL)
- **Настройка аналитики (ДО старта контента):**
  - GSC (Google Search Console) — подтвердить домен
  - Яндекс.Вебмастер — подтвердить домен
  - Яндекс.Метрика — счётчик, цели, Вебвизор
  - GA4 (хотя бы базово)
  - UTM-разметка для внешних площадок

### Фаза 1: JTBD-аудит аудитории (день 1)

1. Загружаем `skill-audience-jtbd-ru`
2. Описываем JTBD-карту для каждого оффера:
   - Ситуация (когда человек ищет)
   - Триггер (что сломалось)
   - Старое решение (что использует сейчас)
   - Новая цель (чего хочет)
   - Барьеры (почему не переключается)
3. Карта четырёх сил: Push / Pull / Anxiety / Habit
4. Лестница осознанности: на какой ступени ЦА по каждому офферу
5. Фразы клиента (из интервью, чатов, отзывов) — это золото для ключей

### Фаза 2: Генерация ключей (день 1-2)

**Методология: 7 пучков на каждый оффер:**

| # | Пучок | Тип формулировок | Пример |
|---|-------|-----------------|--------|
| 1 | Прямой запрос услуги | Формальные, B2B | «автоматизация продаж», «внедрение ии агентов» |
| 2 | Боль / проблема | Живые, через «не» | «менеджеры не ведут crm», «заявки теряются» |
| 3 | JTBD / задача | «как + действие + результат» | «как не терять заявки», «как автоматизировать продажи» |
| 4 | Инструментальные | CRM, платформы, сервисы | «crm для отдела продаж», «телеграм бот для продаж» |
| 5 | Сравнение / выбор | «или», «что лучше», «какой» | «crm или таблица для продаж», «amocrm или битрикс24» |
| 6 | Коммерческие | Цена, подрядчик, под ключ | «внедрение crm цена», «автоматизация продаж под ключ» |
| 7 | Сленг / кривые / низкочастотка | Опечатки, жаргон | «ии для мопов», «чат жпт для продаж», «нейронка» |

**Правила генерации:**
- 20-40 вариантов на каждый пучок
- Используй синонимы: продажи/заявки/лиды/клиенты, CRM/воронка/отдел продаж
- Разные уровни осознанности: «у меня проблема» → «ищу способ» → «выбираю инструмент» → «ищу подрядчика»
- Кривые написания, опечатки, английские вкрапления
- Эмоциональные формулировки — отдельно (для заголовков, не для ядра)

### Фаза 3: Настройка Wordstat API (день 2)

1. **Аккаунт:** Yandex Cloud AI Studio → платёжный аккаунт + привязанная карта
2. **API-ключ:** AI Studio → «Создать API-ключ» → формат `AQVN...`
3. **folderId:** навести на название каталога в AI Studio → скопировать ID

```
POST https://searchapi.api.cloud.yandex.net/v2/wordstat/topRequests
Authorization: Api-Key AQVN...
Content-Type: application/json
Body: {"phrase": "...", "numPhrases": 5, "folderId": "b1g..."}
```

**Квоты (КРИТИЧНО):**
- Wordstat: 10 RPS, 100 запросов в час
- Search (SERP): 10 RPS, 10 000 в час
- Цена Wordstat GetTop: 20 ₽ / 1000 запросов
- Цена Search: 488 ₽ / 1000 запросов

### Фаза 4: Ручной Вордстат (день 2-3)

| Ситуация | Ручной | API |
|----------|--------|-----|
| Первичный сбор идей | ✅ — видишь левую/правую колонки | ❌ |
| Проверка частоток по готовому списку | ❌ — медленно | ✅ |
| Поиск неожиданных ассоциаций | ✅ — правая колонка | ⚠️ |
| Работа с операторами (!, +, []) | ✅ — точные вхождения | ❌ |
| Массовая проверка >50 ключей | ❌ | ✅ |

### Фаза 5: Фильтрация и классификация (день 3-4)

**Что отсекаем (мусор):**
- Слишком широкие
- Чужой интент: 1С-товары, розница, вакансии, курсы
- Инструменты-конкуренты
- Частотка <15

**Матрица распределения:**

| Интент | Тип страницы |
|--------|-------------|
| Зонтичный / позиционный | **main** (главная) |
| Коммерческий («ищу подрядчика») | **landing** (лендинг) |
| Problem-aware («у меня проблема») | **article** (статья) |
| JTBD («как сделать X») | **article** (статья) |
| Инструментальный | **faq** (FAQ-блок) |
| Сравнение | **article** (статья) |
| Отраслевой | **industry** (позже) |
| Эмоциональный | **hook** (в заголовки) |
| Широкий без B2B-интента | **garbage** |

### Фаза 6: API-пробивка частоток (день 3-4)

Используй `scripts/batch-seeds.py`. Ключ из `.wordstat-key`, folderId из AI Studio.
Сравнение с ручным Вордстатом — расхождение должно быть <15%.

### Фаза 7: SERP-анализ (день 4-5)

По каждому коммерческому ключу: **какую страницу хочет видеть поисковик?**

Методы:
- **A: Ручной** (бесплатно) — инкогнито Яндекс + Google, топ-5 результатов
- **B: Yandex Search API** (платно) — `POST /v2/web/search`, ответ в Base64
- **C: ParallelAI** — структура страниц конкурентов
- **D: Perplexity (MCP)** — deep research по конкурентам

**Ключевой вопрос:** если по запросу в топе только статьи — не делай лендинг. Делай статью и веди её на лендинг.

### Фаза 8: Конкурентный разбор (день 5-6)

**SERP-конкуренты** (5-10 шт.) — кто в топе по нашим ключам.
**PR-конкуренты** (3-5 шт.) — топ-агентства в нише, могут не быть в выдаче.

Инструменты: ParallelAI, Gemini, Perplexity, OpenSEO + DataForSEO.

**Что ищем:** офферы, H1/title формулировки, блоки доверия, слабые места, форматы контента.

### Фаза 9: Мини-бриф страницы (день 6-7)

**Шаблон брифа:**

| Поле | Что писать |
|------|-----------|
| Оффер | Что продаём (1 предложение) |
| ЦА | Кому (1 предложение) |
| Ситуация | В каком состоянии клиент сейчас |
| Боль | Что у него не работает |
| Желаемый результат | Что он хочет получить |
| Механизм | За счёт чего мы это делаем |
| Что входит | Конкретные deliverables |
| Доказательства | Кейсы, разборы, цифры, скрины |
| Возражения | Дорого, долго, не сработает, уже пробовали |
| CTA | Что человек должен сделать после ленда |
| SEO-кластер | Под какие запросы страница |
| Связанные страницы | Куда ведём внутренними ссылками |

**Структура лендинга (service page):**
```
H1 (основной запрос)
→ Первый экран (боль + решение)
→ Кому подходит / не подходит
→ Что входит (deliverables)
→ Процесс (этапы работы)
→ Доказательства (кейсы/цифры)
→ Цена / формат работы
→ FAQ (закрыть возражения)
→ CTA (диагностика / заявка)
```

**Структура статьи:**
```
H1 (проблема или JTBD)
→ Короткий ответ в начале (что делать)
→ Контекст проблемы (почему это важно)
→ Разбор причин (почему так происходит)
→ Пошаговое решение
→ Пример / кейс
→ Ошибки (что не работает)
→ Когда нужен подрядчик (сигнал к CTA)
→ FAQ
→ CTA (на лендинг / лид-магнит / аудит)
```

**Источники фактуры (по приоритету):**
1. Оффер и ЦА
2. Боль клиента (реальные формулировки)
3. Механизм (как решаем)
4. Доказательства
5. SEO-интент
6. Конкуренты
7. Редактура

### Фаза 9.5: Производство контента (день 7-10)

1. Пишем черновик по структуре из брифа
2. Интегрируем ключи: H1, H2, body — без переспама
3. Добавляем фактуру: цифры, кейсы, скрины, примеры (без выдумок)
4. Редактируем: убираем воду, канцелярит, AI-штампы (см. SOUL.md)
5. Оптимизируем изображения: alt-текст с ключами, сжатие, WebP
6. Расставляем внутренние ссылки

**AI-политика:**
- AI можно для черновика, но **только с предоставленной фактурой**
- Запрещено: выдумывать кейсы, цифры, клиентов, результаты
- Финальная редактура — всегда человек
- Сверять тон с `SOUL.md`

### Фаза 9.6: Внутренняя перелинковка (день 10-11)

- Silo-структура: группируем страницы по темам, связываем внутри кластера
- Anchor text: не «здесь» и «читать далее», а вхождения ключей страницы-получателя
- Приоритет: главная → лендинги, лендинги → кейсы/аудит, статьи → лендинги/лид-магниты
- 3-5 входящих ссылок на каждую новую страницу
- Контекстные ссылки в теле текста > ссылки в сайдбаре/футере
- Не более 1-2 ссылок на абзац

### Фаза 9.7: E-E-A-T сигналы (день 11-12)

1. **Авторство:** страница автора с фото, bio, экспертизой (Schema: Person)
2. **About-страница:** кто мы, опыт, команда, реквизиты (Schema: Organization)
3. **Внешние упоминания:** собрать ссылки на VC/Habr/Cossa/Spark/отраслевые медиа
4. **Доверие:** кейсы с цифрами, отзывы, сертификаты/награды
5. **Прозрачность:** контакты, ИНН/ОГРН (для Яндекса критично), политика конфиденциальности

### Фаза 10: Техническая упаковка и после публикации

**Перед публикацией:**
- [ ] index/follow (проверить meta robots)
- [ ] canonical правильный
- [ ] Есть в sitemap.xml
- [ ] Нет 404
- [ ] Мобильная версия не сломана
- [ ] PageSpeed (Lighthouse) — зелёная зона или близко
- [ ] Форма работает (цели в Метрике срабатывают)
- [ ] Open Graph (title, description, image, url)
- [ ] Schema: Organization (главная), BreadcrumbList, Article (блог)
- [ ] Внутренние ссылки на страницу (3-5 штук)
- [ ] URL отправлен в GSC / Яндекс.Вебмастер / Bing

**После публикации (GSC-итерации):**
- Неделя 2-4: проверить первые показы
- Неделя 6-8: решить — усиливать или ждать
- Каждую неделю: GSC → запросы с показами → страницы с низким CTR → позиции 8-20

**Сценарии GSC:**

| Ситуация | Проблема | Действие |
|----------|---------|----------|
| Много показов, 0 кликов | Слабый сниппет или мимо интента | Переписать title/description, усилить H1 |
| Позиция 8-20 | Почти зацепилась, не хватает глубины | Добавить H2, FAQ, кейс, внутренние ссылки |
| Не индексируется | Техническая проблема | Проверить noindex, robots, canonical, sitemap |
| Нерелевантные запросы | Текст размазан | Переписать H1/H2, убрать лишние блоки |

### Фаза 10.5: Content Refresh — обновление старого контента

**Когда:** после первой GSC-итерации, когда видно что проседает.

**Что обновлять:**
- Страницы с падающими позициями
- Страницы с показами но без кликов
- Страницы старше 6 месяцев без обновлений
- Страницы где данные устарели

**Как:** добавить H2 под запросы из GSC, обновить цифры/год, добавить свежий кейс, обновить dateModified в schema.

**Периодичность:** раз в квартал — полный аудит; раз в месяц — точечные правки по GSC.

## Hard rules

1. **Не гонись за объёмом ключей.** Лучше 50 релевантных, чем 200 мусорных.
2. **Один интент = одна страница.** Не один ключ = одна страница.
3. **Группируем по интенту, не по словам.**
4. **Сначала ручная чистка, потом платный API.** Не проверяй 5000 ключей через API.
5. **Не насилуй лендинг под инфо-запрос.** Если в топе статьи — делай статью.
6. **Конкуренты ПОСЛЕ семантики, не до.**
7. **Эмоциональные ключи — для заголовков, не для ядра.**
8. **AI-тексты только с фактурой.** Без кейсов, цифр, реальных примеров — нейрослоп.
9. **Главная первая.** Не делай лендинги без главной.
10. **Квота Wordstat: 100/час.** Один поток, пауза 1.2 сек. НИКОГДА параллельные потоки.
11. **Интент определяет тип страницы.** Если по коммерческому ключу в топе только статьи — делаем статью, не лендинг.
12. **Финальная редактура — всегда человек.** AI только для черновика с фактурой.
